# QAMule Project Migration

Use this skill to update a user project created with an older QAMule plugin version so it works with the current plugin.

The migration should preserve user-authored knowledge, tests, trajectories, and local project conventions. Make small, reviewable edits and validate after each major category when possible.

## Version Signals

Treat a project as old-format when one or more of these are present:

- `kb/` exists and `knowledge-base/` does not.
- `dataset/` exists and `trajectory/` does not.
- `pyproject.toml` or lock files mention `pytest-live-pause`, `pytest-u2device`, or `uiautomator2-cli`.
- Tests, docs, agents, or prompts mention old QAMule skills: `kb`, `pytest`, `dataset`, `init`, or `live-pause-failure-triage`.
- Pytest workflows mention `pytest-live-pause watch`, `pytest-live-pause resume`, `run_id`, or `--pause-checkpoint-mock`.

## Current Names

| Old | Current | Notes |
|---|---|---|
| `kb/` | `knowledge-base/` | New QA agent discovers `knowledge-base/**/*.md` by frontmatter. |
| `dataset/` | `trajectory/` | Current CLI is `qamule-trajectory`. |
| `kb` skill | `knowledge-base` skill | Same purpose, less rigid file structure. |
| `dataset` skill | `trajectory` skill | Same purpose, updated CLI. |
| `pytest` skill | `pytest-qamule` skill | Current runtime comes from `pytest-qamule`. |
| `live-pause-failure-triage` skill | `problem-triage` skill | Current triage workflow is generalized. |
| `init` skill | `setup` skill | Current setup installs `pytest-qamule`. |
| `pytest-live-pause` CLI | `qamule pause` / `qamule report` | Use `QAMule Session ID` instead of `run_id`. |

## Migration Workflow

### 1. Explain What Changed

Before editing, first explain to the user what changed in the current QAMule version compared with the old project format. Keep it specific to the detected project instead of listing every possible change.

Use these current-version changes as the baseline:

- The persistent testing knowledge directory changed from `kb/` to `knowledge-base/`.
- Knowledge files are discovered by flat frontmatter in `knowledge-base/**/*.md` instead of relying on only the old `kb/app/_app.md` index convention, so old KB content should be reorganized around reusable overview, screen, flow, dependency, and quirk knowledge files.
- VLM training data moved from `dataset/` to `trajectory/` and is managed by `qamule-trajectory`.
- Pytest runtime integration moved from `pytest-live-pause` / `pytest-u2device` to `pytest-qamule`.
- Pause monitoring now uses `qamule pause watch --session-id <session-id>` and report commands use `qamule report ...`.
- Failure triage is now handled by the general `problem-triage` skill instead of the old `live-pause-failure-triage` skill.
- Project setup now uses the `setup` skill and installs `pytest-qamule`.

Then state the local update plan in concrete terms, for example:

- which directories will be renamed or merged;
- how old `kb/` content will be recomposed into the current `knowledge-base/` structure and frontmatter conventions;
- which dependencies and commands will be updated;
- which tests or docs may need fixture and CLI updates;
- which memory, instruction, prompt, and agent customization files will be scanned for stale QAMule references.

Do not edit until the plan is anchored in the files actually present in the user's project.

### 2. Inspect First

Before editing, check only the local migration surface:

```bash
find . -maxdepth 4 -type d \( -name kb -o -name knowledge-base -o -name dataset -o -name trajectory -o -name tests -o -name .github -o -name .claude -o -name .agents \)
rg -n "pytest-live-pause|pytest-u2device|uiautomator2-cli|pause-checkpoint|live_pause|\bkb/|\bdataset/|\bkb\b|\bdataset\b|live-pause-failure-triage|QAMule|qamule" .
```

Use the results to decide which sections below apply. Do not rewrite unrelated project files.

### 3. Scan Memory And Instructions

Scan user and project customization surfaces before editing project content. Old QAMule instructions can override the new plugin behavior even after files are migrated.

Check these locations when they exist:

```txt
AGENTS.md
CLAUDE.md
.github/copilot-instructions.md
.github/instructions/**/*.instructions.md
.github/prompts/**/*.prompt.md
.github/agents/**/*.agent.md
.claude/**/*.md
.agents/**/*.md
```

Also inspect user-level customization or memory locations exposed by the current agent environment, for example:

```txt
/memories/
/memories/repo/
/memories/session/
${VSCODE_USER_PROMPTS_FOLDER}/
```

When scanning these files, look for stale QAMule references such as:

- old paths: `kb/`, `dataset/`;
- old skills: `kb`, `dataset`, `pytest`, `init`, `live-pause-failure-triage`;
- old packages: `pytest-live-pause`, `pytest-u2device`, `uiautomator2-cli`;
- old commands: `pytest-live-pause watch`, `pytest-live-pause resume`, `--pause-checkpoint-mock`;
- old fixtures or APIs: `live_pause.checkpoint`, `result.reason`.

Update only QAMule-specific stale references. Preserve unrelated user preferences, team instructions, project conventions, and historical notes unless they actively tell the agent to use old QAMule behavior.

### 4. Update Dependencies

For UV projects, migrate the runtime dependencies to the current setup baseline:

```bash
uv remove pytest-live-pause pytest-u2device uiautomator2-cli
uv add pytest "pytest-qamule>=0.2.0" "uiautomator2>=3.7.0" "qamule-trajectory>=0.1.0"
uv sync
```

If the project uses another dependency manager, make the equivalent dependency edits in that manager's native files.

Do not remove unrelated testing, reporting, app-specific, or helper dependencies just because they were installed near old QAMule packages.

### 5. Reorganize Knowledge Base

Rename or merge `kb/` into `knowledge-base/`, then recombine the old content according to the current `knowledge-base` conventions. This is a content migration, not only a directory rename.

```txt
kb/app/_app.md              -> knowledge-base/app/overview.md
kb/app/deps.md              -> knowledge-base/app/deps.md
kb/app/screens/*.md         -> knowledge-base/app/screens/*.md
kb/app/flows/*.md           -> knowledge-base/app/flows/*.md
kb/deps/{package}/*.md      -> knowledge-base/deps/{package}/*.md
```

If `knowledge-base/` already exists, merge file by file. Preserve newer user content and resolve duplicates by keeping both only when they describe different confirmed facts.

Use the new knowledge-base structure as the target shape:

```txt
knowledge-base/
    app/
        overview.md
        screens/
        flows/
    deps/{package}/
```

Recompose old KB content with these rules:

- Move app-wide scope, package name, reset steps, major screen and flow indexes, global prerequisites, and cross-feature quirks into `knowledge-base/app/overview.md`.
- Keep one screen-focused file per stable screen under `knowledge-base/app/screens/`; split old files that mix unrelated screens.
- Keep one flow-focused file per reusable business or navigation flow under `knowledge-base/app/flows/`; split old files that mix independent flows.
- Move system, WebView, account provider, permission controller, chooser, and third-party app knowledge into `knowledge-base/deps/{package}/` instead of main app files.
- Keep quirks inside the most relevant overview, screen, or flow file unless the quirk affects many areas and deserves its own file.
- Remove or merge stale index-only files after their useful content has been carried into overview, screen, flow, or dependency files.
- Preserve confirmed selectors, navigation paths, preconditions, expected results, error recovery paths, ROM or Android-version notes, and dependency ownership facts.
- Do not preserve guesses, one-off run noise, outdated failed attempts, or old TODOs unless they still describe a confirmed reusable constraint.

Every discovered knowledge file must start with flat, single-line YAML frontmatter:

```md
---
name: enable_night_mode
description: "Use when testing night mode, dark theme, appearance settings, or theme switching."
---
```

Conversion rules:

- Preserve the old Markdown body unless it is clearly stale or duplicated.
- Add frontmatter to old files that only had headings and tables.
- Keep `name` stable, lowercase, and identifier-like. Derive it from the path when needed, for example `app_overview`, `screen_settings_home`, `flow_login`, or `dependency_com_android_permission_controller`.
- Write `description` as a trigger sentence starting with `Use when ...` so the SessionStart hook can expose useful knowledge to the agent.
- Keep frontmatter flat. Do not use lists, nested maps, multi-line strings, or unquoted colons.
- Optional fields such as `type`, `scope`, `package`, or `tags` are fine only when they are single-line `key: value` pairs.

Old `kb/app/_app.md` index tables are no longer mandatory, but they are often useful. Keep them in `knowledge-base/app/overview.md` if they remain accurate.

### 6. Migrate Trajectory Data

Rename or merge `dataset/` into `trajectory/`.

Old session contents are generally reusable when each session contains:

```txt
trajectory/{session_dir}/
  step_001.jpg
  trajectory.json
```

Update docs and commands from old script paths to the current CLI:

```bash
uv run qamule-trajectory validate trajectory
uv run qamule-trajectory view trajectory
```

When creating new sessions, use:

```bash
uv run qamule-trajectory init trajectory/<task_slug>_<timestamp> \
  --instruction "Human task instruction here" \
  --app com.example.app \
  --device-model "Pixel 8" \
  --resolution 1080 2400 \
  --android 14
```

Do not rename screenshot files or edit `trajectory.json` unless validation reports a concrete schema problem.

### 7. Update Pytest Runtime Usage

Replace old live pause terminology with the current `pytest-qamule` workflow.

Old command patterns:

```bash
uv run pytest-live-pause watch --run-id=<run_id>
uv run pytest-live-pause resume <pause_id> --result true --reason "..."
uv run pytest-live-pause resume <pause_id> --failure-reason "..."
```

Current command patterns:

```bash
uv run qamule pause watch --session-id <session-id>
uv run qamule pause resume <pause_id> --session-id <session-id> --result true --message "checkpoint accepted"
uv run qamule pause resume <pause_id> --session-id <session-id> --message "checked failure state"
uv run qamule pause abort <pause_id> --session-id <session-id> --message "stop run"
```

Other runtime updates:

- Use the printed `QAMule Session ID: ...` instead of `pytest-live-pause: run_id=...`.
- Use `--qamule-checkpoint-mock-result true|false|none` and `--qamule-checkpoint-mock-message "..."` instead of `--pause-checkpoint-mock`.
- Use `qamule report ...` commands for report inspection.
- Keep `--pause-on-failure` only when live failure inspection is intentionally requested.

### 8. Update Checkpoint Code

Old tests may use the `live_pause` fixture:

```python
def test_checkout_summary(live_pause):
    result = live_pause.checkpoint(task="verify app state")
    assert result.result is True
```

Current tests should use the `checkpoint` fixture:

```python
def test_checkout_summary(checkpoint):
    response = checkpoint("verify app state")
    assert response.result is True
```

If the old code reads `result.reason`, migrate it to `response.message` only after checking the current assertion intent.

### 9. Update Skill And Agent References

Update project documentation, custom agents, prompts, and local instructions to use current names:

- `kb skill` -> `knowledge-base`
- `dataset skill` -> `trajectory`
- `pytest skill` -> `pytest-qamule`
- `live-pause-failure-triage skill` -> `problem-triage`
- `init skill` -> `setup`

Do not change ordinary prose that refers to a product knowledge base unless it is a QAMule path, skill, or command reference.

## Validation

Run the narrowest checks available for the migrated surface.

### Knowledge Base Frontmatter

Use this check to confirm every knowledge file has simple top-level frontmatter:

```bash
uv run python - <<'PY'
import pathlib
import re

root = pathlib.Path("knowledge-base")
pattern = re.compile(r"\A---\s*\n(.*?)\n---(?:\n|\Z)", re.S)
bad = []
for path in sorted(root.rglob("*.md")) if root.exists() else []:
    text = path.read_text(encoding="utf-8")
    match = pattern.match(text)
    if not match:
        bad.append(f"{path}: missing frontmatter")
        continue
    values = {}
    for line in match.group(1).splitlines():
        if ":" not in line or line[:1] in (" ", "\t") or line.lstrip().startswith(("#", "-")):
            continue
        key, value = line.split(":", 1)
        values[key.strip()] = value.strip().strip('"')
    if not values.get("name"):
        bad.append(f"{path}: missing name")
    if not values.get("description"):
        bad.append(f"{path}: missing description")
if bad:
    print("\n".join(bad))
    raise SystemExit(1)
print("knowledge-base frontmatter ok")
PY
```

### Trajectory

```bash
uv run qamule-trajectory validate trajectory
```

Skip this check when the project has no trajectory data.

### Pytest

```bash
uv run pytest --collect-only
```

Run a narrower test path when the migration touched a specific test module.

### Residual Old References

After edits, search for old runtime and path references:

```bash
rg -n "pytest-live-pause|pytest-u2device|uiautomator2-cli|pause-checkpoint|live_pause|\bkb/|\bdataset/|live-pause-failure-triage" .
```

Some historical notes may remain intentionally. Confirm each remaining hit is either harmless history or still needs migration.
